# PHASE 0: PROBLEM STATEMENT DECONSTRUCTION (15-30 min)

---

### **Step 0.1: Deep Reading & Annotation**

Read the problem statement **3 times**:

1. **First pass**: Skim for general understanding
2. **Second pass**: Highlight key terms, constraints, and requirements
3. **Third pass**: Identify hidden expectations and evaluation criteria

### **Step 0.2: Extract Core Components**

Create a structured breakdown:

```markdown
## PROBLEM DECOMPOSITION TEMPLATE

### Explicit Requirements
- [ ] What MUST be built? (non-negotiable features)
- [ ] What technologies are mandated?
- [ ] What constraints exist? (time, tech stack, scope)
- [ ] What data sources are specified?

### Implicit Expectations
- [ ] What's the END USER persona?
- [ ] What's the CORE PAIN POINT?
- [ ] What EXPERIENCE are they expecting?
- [ ] What would make judges say "WOW"?

### Success Criteria (from problem statement)
- [ ] Innovation: _______
- [ ] Creativity: _______
- [ ] Presentation/UI: _______
- [ ] Usability/UX: _______
- [ ] Impact/Benefits: _______
- [ ] Technical Complexity: _______

### Red Flags & Constraints
- [ ] What could disqualify us?
- [ ] What's explicitly forbidden?
- [ ] What assumptions are dangerous?

```

### **Step 0.3: Identify the Problem "Type"**

Categorize the problem to activate appropriate mental models:

| **Type** | **Indicators** | **Focus Areas** |
| --- | --- | --- |
| **Data-Driven** | "analytics", "insights", "dashboard" | Visualization, ML, data pipeline |
| **Automation** | "streamline", "automate", "efficiency" | Workflow, integration, time-saving |
| **Consumer App** | "students", "users", "experience" | UX, engagement, delight factors |
| **Enterprise Tool** | "business", "productivity", "teams" | Reliability, scalability, ROI |
| **Social Impact** | "community", "sustainability", "access" | Impact metrics, storytelling, ethics |
| **Technical Challenge** | "algorithm", "optimize", "performance" | Benchmarks, complexity, innovation |

## üí° PHASE 2: INNOVATION IDEATION (45-90 min)

### **Step 2.1: Divergent Thinking Session**

Generate 20+ ideas using structured prompts:

```markdown
## IDEA GENERATION PROMPTS

### Inversion Thinking
- What if we did the OPPOSITE of standard approaches?
- What if we removed the most "essential" feature?

### Constraint Removal
- What if cost wasn't a concern?
- What if we had perfect data?
- What if users were experts?

### Cross-Domain Inspiration
- How does [different industry] solve similar problems?
- What if we applied [gaming/social media/finance] mechanics?

### Technology Forcing
- How would we solve this with [AI/VR/blockchain/voice]?
- What if the interface was [invisible/ambient/wearable]?

### User Transformation
- What if the user was [10 years old / 80 years old / expert]?
- What if they were using this [in a car / in bed / while cooking]?

### Extreme Scenarios
- What if we had to deliver value in 30 seconds?
- What if we could only use 3 screens?
- What if the user was offline?

```

### **Step 2.2: Convergent Filtering**

Score each idea on the **DICE Framework**:

```markdown
## DICE SCORING (1-10 scale)

D - Differentiation
- How unique is this compared to existing solutions?

I - Impact
- How significantly does this improve the user's life?

C - Credibility
- Can we actually build this in the timeframe?

E - Excitement
- Will judges/users say "I need this NOW"?

TOTAL SCORE: ___ / 40

‚úÖ Score 30+: Core feature
‚ö†Ô∏è Score 20-29: Nice-to-have
‚ùå Score <20: Cut or defer

```

### **Step 2.3: Innovation Thesis Formulation**

Articulate your unique insight in one sentence:

```markdown
## INNOVATION THESIS TEMPLATE

"While everyone else is building [CONVENTIONAL APPROACH],
we realized that users actually need [INSIGHT],
so we're creating [UNIQUE SOLUTION],
which delivers [SPECIFIC BENEFIT]
in a way that's [DIFFERENTIATOR]."

Example:
"While everyone else is building meal planning apps that require
extensive manual input, we realized that users actually need
instant value with zero effort, so we're creating a photo-based
pantry assistant with AI personality, which delivers personalized
recipes in 10 seconds in a way that's emotionally engaging and fun."

```

### **Step 9.2: Pivot Decision Framework**

```markdown
## WHEN TO PIVOT

### Red Flags (consider pivoting if):
- Core technology isn't working after 25% of time spent
- User value is unclear or weak
- Team morale is very low
- Scope is obviously unachievable

### Pivot Decision Matrix

| Situation | Pivot Strategy |
|-----------|----------------|
| Tech not working | Simplify approach, use different tech |
| Too ambitious | Cut features, focus on fewer flows |
| Unclear value | Refocus on strongest use case |
| Weak differentiation | Add unique twist or change angle |

### Pivot Protocol
1. Call team huddle
2. Assess: What's working? What's not?
3. Decide: Adjust or abandon?
4. Commit: New plan documented and shared
5. Execute: Full focus on new direction

```
